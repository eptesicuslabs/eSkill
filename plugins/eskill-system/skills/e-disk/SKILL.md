# Disk Usage Analyzer

This skill analyzes disk usage attributable to development artifacts including dependency directories, Docker images and layers, build caches, virtual environments, and editor caches. It identifies the largest space consumers, classifies them by reclaimability, and generates safe cleanup commands with estimated savings.

## Prerequisites

Confirm the scope of the analysis with the user:

- **Project-scoped**: Analyze only the current project directory.
- **User-scoped**: Analyze all development artifacts under the user's home directory.
- **System-scoped**: Analyze system-wide development tool caches and containers.

Default to project-scoped if not specified. For user-scoped and system-scoped analysis, note that scanning large directory trees may take longer.

## Step 1: Check Overall Disk State

Use `sys_disk` to retrieve filesystem utilization for all mounted volumes.

Record:

| Mount / Drive | Filesystem | Total | Used | Available | Usage % | Status |
|---------------|-----------|-------|------|-----------|---------|--------|
| C: | NTFS | 500 GB | 420 GB | 80 GB | 84% | Warning |
| D: | NTFS | 1 TB | 200 GB | 800 GB | 20% | Healthy |

Status thresholds:

| Usage | Status | Urgency |
|-------|--------|---------|
| Below 70% | Healthy | No action needed |
| 70-85% | Warning | Proactive cleanup recommended |
| 85-95% | Critical | Cleanup needed soon |
| Above 95% | Emergency | Immediate cleanup required |

Identify the volume where the project resides. Focus analysis on that volume. If multiple volumes are critical, note all of them.

## Step 2: Scan Project-Level Artifacts

Use `fs_list` and `shell_exec` to measure the size of common development artifact directories within the project.

| Artifact | Typical Path | Description |
|----------|-------------|-------------|
| Node modules | `node_modules/` | npm/yarn/pnpm dependencies |
| Python venv | `.venv/`, `venv/`, `env/` | Python virtual environments |
| Build output | `dist/`, `build/`, `out/`, `target/` | Compiled or bundled output |
| Coverage reports | `coverage/`, `.nyc_output/`, `htmlcov/` | Test coverage data |
| IDE files | `.idea/`, `.vscode/` (workspace storage) | Editor workspace data |
| Storybook build | `storybook-static/` | Built Storybook output |
| Next.js cache | `.next/` | Next.js build and dev cache |
| Nuxt cache | `.nuxt/`, `.output/` | Nuxt build artifacts |
| Rust target | `target/` | Cargo build output (can be very large) |
| Go cache | Go module and build cache | Managed by `go clean` |
| Gradle cache | `.gradle/`, `build/` | Gradle build output |
| Maven target | `target/` | Maven build output |
| Terraform plugins | `.terraform/` | Provider binaries |
| Git objects | `.git/` | Repository history and pack files |

For each found artifact, use `fs_info` to retrieve its size and metadata. `fs_info` works cross-platform and returns structured data, eliminating the need for platform-specific shell commands. For directories, `fs_info` reports the aggregate size including all nested contents. For individual large files, `fs_info` provides exact byte counts that can be used for precise reporting.

Fall back to `shell_exec` with platform-specific commands (`du -sh` on Linux/macOS, `Get-ChildItem -Recurse | Measure-Object` on Windows) only if `fs_info` does not support recursive directory sizing on the current eMCP version.

Record findings:

| Artifact | Path | Size | Reclaimable | Impact of Deletion |
|----------|------|------|-------------|-------------------|
| node_modules | ./node_modules | 850 MB | Yes | Restored by `npm install` |
| .next cache | ./.next | 320 MB | Yes | Rebuilt on next dev/build |
| target (Rust) | ./target | 2.1 GB | Partial | Full rebuild on next `cargo build` |
| .git | ./.git | 150 MB | No (special) | Repository history |
| coverage | ./coverage | 45 MB | Yes | Regenerated by test run |

## Step 3: Scan Docker Artifacts

Use `shell_exec` with Docker CLI to analyze Docker's disk usage.

**Docker system overview**:
Use `docker_images` to list all locally available images with their repository, tag, image ID, size, and creation date. This provides structured data about the image inventory without requiring shell parsing. Supplement with `shell_exec` to run `docker system df` for the full summary including containers, volumes, and build cache:

| Type | Total | Active | Size | Reclaimable |
|------|-------|--------|------|-------------|
| Images | <N> | <N> | <size> | <size> |
| Containers | <N> | <N> | <size> | <size> |
| Volumes | <N> | <N> | <size> | <size> |
| Build Cache | | | <size> | <size> |

**Unused images**: List Docker images not referenced by any container.
- Use the `docker_images` data and cross-reference with `docker_ps` to identify images not backing any container (running or stopped).
- Dangling images (untagged `<none>:<none>`) are always safe to remove.
- Old base image versions (e.g., `node:16` when the project uses `node:20`) are candidates for removal.

**Stopped containers**: List containers in "Exited" state that may still consume disk space through their writable layer and logs.

**Unused volumes**: List Docker volumes not mounted by any container. These often contain old database data from deleted containers.

**Build cache**: Docker's build cache can grow significantly with frequent image builds.

Record the total Docker disk usage and the reclaimable portion.

## Step 4: Scan User-Level Caches (If User-Scoped)

For user-scoped analysis, check common development tool caches.

| Cache | Default Location | Description |
|-------|-----------------|-------------|
| npm cache | `~/.npm/` | Downloaded package tarballs |
| yarn cache | `~/.cache/yarn/` or `~/.yarn/cache/` | Yarn package cache |
| pnpm store | `~/.local/share/pnpm/store/` | pnpm content-addressable store |
| pip cache | `~/.cache/pip/` | Downloaded Python packages |
| cargo registry | `~/.cargo/registry/` | Crate source and index |
| cargo git | `~/.cargo/git/` | Git dependencies |
| Go module cache | `~/go/pkg/mod/` | Downloaded Go modules |
| Gradle cache | `~/.gradle/caches/` | Gradle dependency cache |
| Maven repository | `~/.m2/repository/` | Maven dependency cache |
| Homebrew cache | `~/Library/Caches/Homebrew/` | macOS package downloads |
| NuGet cache | `~/.nuget/packages/` | .NET package cache |
| VS Code extensions | `~/.vscode/extensions/` | Installed extensions |

For each cache found, measure its size and determine:
- Whether it can be safely cleared.
- Whether clearing it will require re-downloading on next use.
- How fast it will regrow.

## Step 5: Identify Duplicate and Redundant Artifacts

Search for patterns that indicate wasted space.

**Duplicate node_modules**: In monorepos or projects with nested dependencies, `node_modules` may appear at multiple levels. Use `fs_search` to find all `node_modules` directories.

**Multiple Python venvs**: Projects may have leftover virtual environments from different Python versions or tool migrations.

**Orphaned build outputs**: Build directories from deleted branches or old configurations.

**Stale lock files**: Multiple lock file types in the same project (e.g., both `package-lock.json` and `yarn.lock`) suggesting a package manager migration where the old lock file was not cleaned up.

**Large files in git**: Use `fs_info` on files within the repository to identify those exceeding 10 MB. `fs_info` provides exact byte counts per file, making it straightforward to rank files by size without parsing shell output. Check for:
- Files over 10 MB in the repository.
- Binary files (images, fonts, PDFs, archives) that could be externalized.
- Database dumps or fixture files committed to the repository.

Record each finding with its exact size from `fs_info` and whether it can be addressed.

## Step 6: Classify and Prioritize Cleanup Targets

Organize all findings by reclaimability and impact.

| Category | Items | Total Size | Reclaimable | Rebuild Cost |
|----------|-------|-----------|-------------|--------------|
| Safe to delete | Dangling Docker images, stopped containers, coverage reports | <size> | 100% | None |
| Rebuildable | node_modules, build output, caches | <size> | 100% | Seconds to minutes |
| Partially reclaimable | Rust target (keep debug, remove release or vice versa), Docker build cache | <size> | 50-80% | Minutes to hours |
| Requires caution | Docker volumes (may contain data), .git (can be repacked) | <size> | Varies | May lose data |
| Do not delete | Source code, configuration, database files | - | 0% | - |

Sort cleanup targets by size descending within each category. The highest-impact cleanup items are large, safe-to-delete artifacts.

## Step 7: Generate Cleanup Commands

For each cleanup target, provide the specific command to reclaim space.

**Docker cleanup**:

| Action | Command | Estimated Savings |
|--------|---------|------------------|
| Remove dangling images | `docker image prune -f` | <size> |
| Remove all unused images | `docker image prune -a -f` | <size> |
| Remove stopped containers | `docker container prune -f` | <size> |
| Remove unused volumes | `docker volume prune -f` | <size> |
| Remove build cache | `docker builder prune -f` | <size> |
| Full Docker cleanup | `docker system prune -a --volumes -f` | <size> |

**Project artifact cleanup**:

| Action | Command | Estimated Savings |
|--------|---------|------------------|
| Remove node_modules | `rm -rf node_modules` | <size> |
| Remove build output | `rm -rf dist build out` | <size> |
| Remove coverage | `rm -rf coverage .nyc_output` | <size> |
| Remove Next.js cache | `rm -rf .next` | <size> |
| Remove Rust debug builds | `cargo clean` | <size> |
| Remove Python venv | `rm -rf .venv` | <size> |
| Remove Terraform providers | `rm -rf .terraform` | <size> |

**Package manager cache cleanup**:

| Action | Command | Estimated Savings |
|--------|---------|------------------|
| Clear npm cache | `npm cache clean --force` | <size> |
| Clear yarn cache | `yarn cache clean` | <size> |
| Clear pip cache | `pip cache purge` | <size> |
| Clear Go module cache | `go clean -modcache` | <size> |
| Clear Gradle cache | `rm -rf ~/.gradle/caches/` | <size> |

**Git maintenance**:

| Action | Command | Estimated Savings |
|--------|---------|------------------|
| Garbage collection | `git gc --aggressive` | Varies |
| Remove stale remote refs | `git remote prune origin` | Minimal |

Present commands grouped by risk level (safe, moderate, use caution).

## Step 8: Execute Cleanup (With Confirmation)

Before executing any cleanup command, present the plan to the user:

```
## Cleanup Plan

### Safe Deletions (no data loss risk)
1. Docker dangling images: ~500 MB
2. Coverage reports: ~45 MB
3. Build output (dist/): ~120 MB

### Rebuildable Deletions (require rebuild)
4. node_modules: ~850 MB (restored by `npm install`)
5. .next cache: ~320 MB (rebuilt on next dev)

### Total Estimated Savings: ~1.8 GB

Proceed with all? Proceed with safe only? Select specific items?
```

Execute only after explicit confirmation. After each cleanup operation, verify the space was reclaimed by re-checking the filesystem.

## Step 9: Verify Cleanup Results

After cleanup, measure the results.

Use `sys_disk` to compare disk state before and after:

| Metric | Before | After | Reclaimed |
|--------|--------|-------|-----------|
| Volume usage | 84% (420 GB) | 78% (390 GB) | 30 GB |
| Project directory | 3.5 GB | 800 MB | 2.7 GB |
| Docker system | 15 GB | 3 GB | 12 GB |

Report the total space reclaimed and the new usage percentage.

If the cleanup was insufficient to resolve the user's space issue, suggest escalation options:
- Analyze other project directories for similar cleanup opportunities.
- Check for large application logs, crash dumps, or system update caches.
- Consider moving large repositories or Docker volumes to a different disk.

## Step 10: Recommend Ongoing Maintenance

Suggest practices to prevent disk space from accumulating over time.

1. **npm/yarn clean installs**: Use `npm ci` in CI/CD instead of `npm install` to avoid accumulating cached packages.
2. **Docker cleanup in CI**: Add `docker system prune -f` to CI pipeline post-steps.
3. **Build artifact cleanup**: Add cleanup targets to `Makefile` or `package.json` scripts:
   ```json
   "scripts": {
     "clean": "rm -rf dist coverage .next node_modules"
   }
   ```
4. **Git maintenance**: Schedule periodic `git gc` via a git hook or system cron.
5. **Disk usage monitoring**: Set up a disk usage alert (system notification when usage exceeds 80%).
6. **Docker volume management**: Name Docker volumes explicitly and document their purpose. Unnamed volumes are easily orphaned.
7. **Dependency pruning**: Regularly run `npm prune`, `pip check`, or equivalent to remove unused installed packages.

If the project has a `Makefile` or task runner, offer to add a `clean` target that performs the common cleanup operations.

## Edge Cases

- **Monorepos with many node_modules**: Monorepos managed by npm workspaces, yarn workspaces, or pnpm may have `node_modules` at the root and in each package. The total can be several GB. Use `pnpm` (content-addressable store) as a space-saving recommendation.
- **Rust projects**: `target/` directories in Rust projects can exceed 10 GB, especially with multiple build profiles (debug, release) and many dependencies. `cargo clean` removes everything; suggest `cargo clean --release` or `cargo clean --profile dev` for selective cleanup.
- **Shared Docker cache**: On machines used for multiple projects, Docker images may be shared. Aggressive pruning with `docker image prune -a` removes images used by other projects. Warn the user if other projects' images are detected.
- **Network-mounted volumes**: If the project or Docker volumes are on a network mount (NFS, SMB), disk usage analysis may be slow or report differently than local storage. Note the volume type.
- **WSL disk space**: On Windows with WSL, the WSL virtual disk (`ext4.vhdx`) grows but does not shrink automatically. After cleanup within WSL, the user may need to compact the VHDX file to reclaim space on the Windows host.
- **CI/CD runners**: For self-hosted CI runners, disk cleanup is critical. Docker images and build artifacts from many branches accumulate rapidly. Recommend automated cleanup policies.
